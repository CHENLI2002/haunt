# Fan Client · Conventions

## 0. Form-factor decision (important, easy to get wrong)

- **v0 uses Web / PWA**, not a native App (decision `docs/venue-first-tech-plan.md` §11-3).
  - Why: **get the validation signal as fast as possible** — the one thing v0 must prove is "will fans follow a venue".
  - The v0 client can share the stack with the public pages (Next.js); implement follow/feed directly on the Web; PWA push optional.
- **The native Expo App is pushed to v1.** The Expo conventions below are for v1; don't slow validation in v0 just to ship a native App.

## 1. Expo / React Native conventions (v1)

- **Expo (managed) + React Native**, one codebase for iOS + Android, iteration first.
- Navigation uses **expo-router** (file-based routing, same mental model as Next) or React Navigation — pick one across the whole project, don't mix.
- Components use shared design tokens (see [design-system](../design-system/SKILL.md)) for visual consistency with the public pages.
- Share **TypeScript types** across front/back (the main payoff of choosing TS, decision §11-1) — DB types are generated by Supabase and reused on the client; don't hand-copy types.

## 2. Connecting to Supabase

- Use `@supabase/supabase-js`, with the **anon key only**; **never** put the service key in the client.
- All data access is constrained by **RLS** (see [data-model](../data-model/SKILL.md) §4): the client can only get data the policy allows. Security relies on RLS, not on the client.
- Auth uses **Supabase Auth**; persist the session + auto-refresh tokens.

## 3. Push notifications

- Use **Expo Notifications** (APNs / FCM underneath).
- Registration flow: get the Expo push token → store it on the user record → on follow→push fanout, push by token.
- Fanout logic: when a followed venue gets a new event, notify followers. At local scale this is **one query, no message queue** (see §4, §8.5) — don't reach for Kafka/SQS.
- v0 (Web/PWA) can use Web Push, or skip push initially and put the validation signal first.

## 4. Core screens (aligned to feature versions)

- v0: login, venue detail, **follow**, follow-feed. Instrument follow behavior (self-hosted PostHog).
- v1: artist detail, "where do they play next", IG-synced media.
- v2: live notes (one line, ≤140, not a rating), regulars circle.

## 5. Do not

- ❌ Don't ship native in v0 for "App experience" — violates §11-3's validation-first priority.
- ❌ Don't make client-side permission checks a security boundary — RLS is the boundary.
- ❌ Don't hand-maintain types that duplicate the backend — use the generated TS types.
